# Knowledge Pack Specification

This document defines the canonical Knowledge Pack format, layout, manifest schema, metadata, assets, checksums, and compatibility rules.

## Pack Purpose

A Knowledge Pack is an immutable, self-contained artifact that packages validated knowledge artifacts, metadata, schemas, and provenance sufficient for Eunoia AI OS to consume without further transformation.

## Directory Layout (canonical)

```
/<packId>-<version>/
  /manifests/
    manifest.json
    manifest.sig
  /content/
    articles/
    schemas/
    assets/
  /metadata/
    provenance.json
    taxonomy.json
  /checksums/
    checksums.sha256
  /license/
    license.txt
  README.md
```

## Manifest (manifest.json)

Required fields:
- `packId` — stable identifier.
- `version` — semantic version.
- `createdAt` — ISO timestamp.
- `components` — list of packaged artifacts with paths, content type, checksum, and schemaRef.
- `provenance` — references to Source/Generation run/author.
- `compatibility` — supported consumer contract versions (major.minor).

Signing: `manifest.sig` is a detached signature over `manifest.json` generated by the Publisher using a canonical signature algorithm.

## Metadata and Provenance

`provenance.json` must include resolver versions, registry references, generation `runId`, and identity of the actor who initiated the build.

## Checksums and Integrity

All content files must be listed in `checksums/checksums.sha256` using canonical lexicographic ordering. The Pack Builder computes a pack-level `packChecksum` that is appended to the registry record.

## Asset Types and Constraints

- Allowed asset MIME types must be explicit in `manifest.json`.
- Binary assets must have accompanying textual metadata describing licensing and processing notes.

## Compatibility & Versioning

- Packs must declare `compatibility.minimum` and `compatibility.maximum` consumer contract versions.
- Minor pack bumps indicate non-breaking additions; major bumps indicate breaking changes.

## Error Conditions

- Missing required manifest fields: reject pack.
- Checksum mismatch: reject and record incident.
- Unsupported asset type: fail validation with remediation guidance.

## Extension Mechanisms

- `manifest.extensions` permits namespaced fields for monetization, region-specific metadata, and publisher-defined annotations. Consumers must ignore unknown extension namespaces.

---
_The Knowledge Pack format balances strict verification with future extensibility to support marketplace capabilities._

## Monetization & Licensing Fields

- `manifest.monetization` (optional): describes commercial terms and entitlements. Minimal fields:
  - `licenseType` (e.g., `free`, `paid`, `enterprise`)
  - `price` (decimal, optional) and `currency` (ISO 4217)
  - `entitlementId` (for downstream license checks)
  - `billingMetadata` (opaque map for billing systems)

- `manifest.license` must link to `license/license.txt` and include a machine-readable `licenseRef` for enforcement.

## Key Discovery & Trust

- The Pack Registry MUST publish publisher public key metadata for signature verification. `manifest.signer` points to a `signerId` resolvable in the Pack Registry.

## Lifecycle & Retention

- Packs must include optional `retention` metadata to indicate recommended archival policies and regional constraints.
